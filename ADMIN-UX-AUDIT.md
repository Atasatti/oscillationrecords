# Admin Backend — UX / Workflow Audit

_Scope: every flow under `/admin` (shell, dashboard, live data, releases list + creation + editor, artists, homepage hub, subscribers, settings/site-content) plus a cross-cutting consistency pass._

## How this was produced

A multi-agent audit: 11 flows were each deep-read by a dedicated reviewer against an
11-dimension UX rubric (IA/navigation, task friction, forms & validation, feedback & system
status, empty states, error handling, destructive actions, consistency, accessibility,
responsive/touch, perceived performance). Every finding was then **adversarially verified
against the actual source** by a second agent that defaulted to "not real" unless it could
confirm the problem in code.

- **162** raw findings → **8 refuted and dropped** (misreads — see Appendix B) → **154 real**.
- 99 confirmed by the automated verifier; the verifier for 3 flows (dashboard, release-editor,
  artists) was cut short by a rate limit, so their **critical/high findings were re-verified by
  hand against source** for this report. Two were recalibrated down (noted inline).

**Severity tally (154):** 1 critical · 7 high · 38 medium · 89 low · 19 nit.

> The most useful takeaway is not the 154 line-items — it's the **handful of cross-cutting
> patterns** below. Fixing those root causes resolves a large share of the individual findings
> at once.

---

## Executive summary

This is a **well-built admin** with a genuinely strong foundation: a CVA-based design system
(Button/Badge/Dialog), a reusable `PageHeader`, correct longest-prefix sidebar active state,
breadcrumbs, optimistic inline toggles with rollback, skeleton list loaders, confirm dialogs on
every delete, a background upload queue with per-file progress/retry, and thoughtful empty-state
copy. The list pages, the artists flow, and the dashboard are especially polished.

The weaknesses are concentrated and consistent. They cluster into a few root causes — **silent
data loss, divergent save/feedback models, an ordering data-model collision, and modal/combobox
accessibility** — and the catalog visibly splits into "two products": the list pages + artist
editor follow the design system, while the release editor and release detail page roll their own
headings, raw Tailwind grays, and a different save model.

### Top priorities (highest impact-to-effort)

1. **Stop the release editor from silently deleting saved tracks.** Track autosave + the server's
   "prune anything not in the payload" logic means clearing a saved track's name/artist for a
   moment deletes it permanently, with a green "Saved" shown. _(release-editor-1, CRITICAL)_
2. **Give the two ordering tools separate fields.** "Custom order" and Homepage "Coming Soon" both
   write the same `sortOrder` column over overlapping sets, so curating one silently scrambles the
   other and the live public grid. _(releases-list-1, HIGH)_
3. **Add unsaved-changes guards to the form editors.** Artist editor, release-details, and the
   hero/studio/footer editors all discard everything on Cancel/Back/tab-switch with no warning —
   the single most repeated data-loss footgun. _(crosscut-4 + artists-2, release-editor-3,
   settings-2, homepage-3)_
4. **Don't default new releases to "Publish".** A new release defaults to RELEASED with no track
   gate, so one click puts a track-less release live on the public site. _(release-create-1)_
5. **One small Toast.tsx change fixes ~7 accessibility findings:** make error toasts
   `role="alert"` (assertive), add a dismiss button, and pause-on-hover. _(shell-6/7, crosscut-7,
   artists-5/6, release-editor-11, settings-9)_

## Cross-cutting themes

These recurring patterns are where the leverage is. Each bullet lists the per-flow findings it
absorbs.

### A. Silent data loss — the dominant theme
- **Autosave prune deletes saved tracks** — clearing a saved track's required field + the 800ms
  debounce + server prune = permanent loss. _(release-editor-1 — CRITICAL; verified)_
- **No unsaved-changes guard on any form editor** — Cancel/Back/sidebar-click/tab-switch/refresh
  silently discards dense forms (bios, distributor codes, imported metadata, staged image
  uploads). _(crosscut-4, artists-2, release-editor-3, settings-2, homepage-3, release-create-2)_
- **Autosave failures aren't surfaced** — a failed track save just toasts (4.5s) and stops; no
  persistent "unsaved" state, no retry, not covered by the `beforeunload` guard.
  _(release-editor-2)_
- **Server silently drops/transforms input** with a success message — biography required server-
  side but not client-side; footer URLs without a scheme nulled; invalid contact email nulled;
  blank image can null an existing photo. _(artists-1, settings-4, artists-9, artists-3)_
- **Apply-to-all / image-remove have no preview, confirm, or undo** and autosave immediately.
  _(release-editor-15, settings-7, crosscut-10, homepage-12)_

### B. Inconsistent save models within a single screen
- Release editor: details are **explicit-save**, the tracklist **autosaves** — opposite mental
  models on one screen, no cue. _(release-editor-4)_
- Homepage hub: **three** models on one screen — autosave-on-drag (carousels), explicit-save
  (hero/photos), plus mixed feedback. _(homepage-4)_
- Settings vs its host page: image editors save via inline text, the same page's release controls
  use toasts. _(settings-5)_
- **Recommendation:** pick one model per interaction type; if a screen must mix, show a loud,
  persistent "Unsaved changes — Save" affordance on the explicit-save half.

### C. Feedback channel & toast inconsistency
- **Toast root issues (one fix, many findings):** all toasts are `role="status"` (polite, so
  errors can be missed), auto-dismiss at 4.5s, no manual close, no pause-on-hover.
  _(shell-6, shell-7, crosscut-7, artists-5, artists-6, release-editor-11, settings-9)_
- **Reorder success is invisible everywhere** — a transient "Saving…" then nothing; failures
  "snap back" which reads as a bug, not an error. _(releases-list-2, releases-list-4, homepage-6,
  artists-16, release-editor-18)_
- **Inline colored text vs toast vs nothing** for the same operation. _(livedata-4, settings-5,
  homepage-5)_

### D. Ordering data model
- **`sortOrder` is shared by two tools over overlapping sets** → silent corruption of curated
  public order. _(releases-list-1 — HIGH; verified)_
- **Two track-reorder surfaces, two APIs, two failure behaviors** (editor snapshot-PATCH vs detail
  page's dedicated endpoint). _(release-editor-5)_
- **Recommendation:** one canonical ordering field per public surface (`comingSoonOrder`), and
  standardize track reorder on the dedicated id-list endpoint (which also de-risks A's prune bug).

### E. Modal & combobox accessibility (hand-rolled where Radix exists)
- **Mobile nav drawer** has no focus trap, no Escape, no focus-return, no `aria-modal` — yet every
  other modal uses Radix Dialog which gives all of this free. _(shell-4, shell-13, crosscut-5,
  homepage-8)_
- **Custom dropdowns aren't keyboard/listbox-operable:** quick search, the primary-artist
  MultiSelect (a required, publish-blocking field), GenrePicker, the add-to-carousel search, and
  the NewReleaseDialog picker. _(shell-5, release-create-6, release-editor-12, artists-14,
  homepage-7)_
- **Recommendation:** rebuild the drawer on Radix Dialog; introduce one shared combobox primitive.

### F. Loading / refetch blanks content
- **Period switch blanks the whole dashboard** to a spinner, losing scroll + context.
  _(dashboard-1 — HIGH)_
- **A single failed background poll wipes the Live data page.** _(livedata-2)_
- **Partial fetch failures silently hide whole sections** (drill-down shows "No data"; Conversion
  block vanishes). _(dashboard-3, dashboard-7)_
- **Full-page spinner instead of skeleton → layout shift** on the most-used screens.
  _(dashboard-2, livedata-1/9, release-editor-16, crosscut-8/9, homepage-10)_

### G. Information architecture & naming
- Detail pages highlight the **wrong** sidebar item; breadcrumb parent links **404**; singular vs
  plural route families for artists; site-content split across "Settings" and "Homepage"; one
  release state called three names (Live/Released/public); breadcrumb "Data" ≠ nav "Live data".
  _(shell-2, shell-3, artists-4, settings-1, crosscut-3, livedata-5, subscribers-10)_

### H. Touch targets < 44px (quick sweep)
- Trash/actions/drag/remove controls are 24–32px in several places. _(subscribers-12,
  releases-list-15, homepage-13, settings-10)_

### I. Empty states are dead-ends
- "No X yet" with no inline CTA where the user is looking. _(releases-list-10, artists-12,
  dashboard-8, settings-14)_

### Subscribers

The Subscribers flow is a tight, well-scoped single-page list with search, pagination, CSV export, and per-row GDPR delete. It is one of the cleaner admin surfaces: debounced search, a proper confirm dialog for the destructive delete, a CSV formula-injection guard on the server, disabled-while-submitting buttons, and sensible loading/empty/error toasts. The main UX gaps are in resilience and responsiveness rather than core flow: the table has no horizontal-overflow handling on mobile, deleting the last row on a page strands the user on an empty page, the CSV export is unbounded and fires with zero feedback (and can be silently popup-blocked), and the empty state is a dead-end that doesn't orient a first-run admin. None are critical; the high-value fixes are small.

It also diverges from the rest of the admin by not using the shared toast system for errors.

### Global shell & navigation

The admin shell has a clean, consistent visual foundation: a persistent desktop rail, a slide-out mobile drawer, breadcrumbs, a reusable PageHeader, and a dependency-free toast system. Active-link logic, breadcrumb assembly, and route-change-driven drawer-close are all thoughtfully built. However, the shell falls short on navigation correctness and accessibility: the "admin-wide" quick search is not actually mounted in the shell (it lives only inside the dashboard body), detail pages mis-highlight the wrong sidebar item, breadcrumbs generate links to routes that 404, and the mobile drawer has no focus trap, Escape handling, or focus-return. These are concrete, fixable gaps rather than architectural problems.

### Releases list & ordering

The releases list is a well-built, professional admin surface: status tabs with live counts, debounced search, optimistic inline toggles, sortable columns, skeleton loading, bulk delete with confirmation, and drag-to-reorder with optimistic rollback. The biggest problem is not visual — it is a data-model collision: the "Custom order" drag panel and the Homepage "Coming Soon" drag list both write the SAME `sortOrder` field over overlapping release sets, so reordering one silently scrambles the other and the public grid. Secondary issues cluster around reorder feedback (no success toast, weak save/disabled state, no error boundary), drag-and-drop accessibility/touch ergonomics, and a few empty-state and consistency gaps. None of the visible interactions are broken for the happy path, but the ordering model and reorder feedback need attention before this is trustworthy for a non-technical label admin.

### crosscut

The admin has a genuinely strong shared foundation — a CVA-based Button/Badge/Dialog system, a reusable PageHeader, a sidebar with correct longest-prefix active state, breadcrumbs, optimistic inline toggles, skeleton list loaders, and confirm dialogs for every delete. The cross-cutting weakness is that the catalog splits into two stylistic worlds: the list pages and ArtistEditor adhere to the design system (PageHeader, semantic color tokens, skeletons), while the release-side editors and detail pages (ReleaseEditor, release detail) roll their own headings (text-4xl instead of the fluid var(--text-h2) scale) and lean on raw Tailwind grays (text-gray-400, text-white, bg-[#141414], border-white/10) instead of the semantic tokens used everywhere else. Terminology for a single release state fractures into three words (Live / Released / public), no form editor warns about unsaved changes, and the hand-rolled mobile drawer skips the focus-trap/escape/focus-return that the Radix Dialog gives every other modal. None of these are individually severe, but together they read as two teams shipping the same product.

### Artists

The artists flow is one of the more polished areas of this admin: a single shared ArtistEditor backs both create and edit (true parity), the list has rich filtering, optimistic visibility/featured toggles with rollback, skeleton loading, debounced search, and a thoughtful set of metadata-enrichment dialogs (Spotify / MusicBrainz / ISNI) that never clobber existing data. The most serious problem is a client/server validation mismatch: the editor only requires a name and image, but the create/update API rejects any artist with an empty biography, surfacing only a blanket toast with no inline field error — a guaranteed dead-end for a first-time user. Secondary issues cluster around feedback (error toasts not announced assertively, no unsaved-changes guard, no dirty/disabled state on the save button when nothing changed) and a few consistency/IA gaps (the "View releases" action points at a different route prefix than the row link, genres entered via a free-text comma string rather than chips). None of these are catastrophic, but the biography mismatch and the missing dirty guard cause real, repeatable user pain.

### Release editor (tracks/credits/uploads)

The Release Editor is an ambitious, mostly well-built flow: background upload queue with per-file progress/retry, debounced track autosave, a publish-readiness gate, and dnd reordering. But the save model has a serious data-loss trap — debounced autosave POSTs a filtered "snapshot" while the server treats the tracks array as authoritative and prunes anything missing, so clearing a field on a saved track can silently delete it. The flow is also split awkwardly across two pages (details edited in the editor, tracks reorderable on both the editor and the detail page using two different APIs), the release-details form has no unsaved-changes guard, and the metadata-lookup APIs the brief flags (Spotify/MusicBrainz/ISNI) are not actually wired into this editor at all. Accessibility and consistency gaps (mixed save patterns, non-keyboard-navigable MultiSelect, error toasts announced as status) round out the issues.
